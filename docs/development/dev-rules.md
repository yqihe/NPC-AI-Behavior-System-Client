# 通用开发规则

## 文档同步

**强制规则：代码改动和文档更新必须在同一步骤完成。**

来源：游戏服务端开发中多次出现代码改了但文档没同步的问题。

## Git 规则

- 每个需求创建 feature 分支：`feature/<name>`
- commit message 格式：`类型(范围): 描述`
  - 类型：`feat` / `fix` / `test` / `refactor` / `docs` / `chore`

## 经验沉淀（从游戏服务端继承）

以下是游戏服务端开发中积累的教训，客户端也适用：

| 教训 | 来源 | 应用到客户端 |
|------|------|-------------|
| nil/null 防护 | 服务端 nil slice → JSON null | 服务端 world_snapshot 的 npcs 字段保证为 `[]` 不为 null，但客户端仍应防御性检查 |
| 协议字段变更 | 服务端多次调整消息结构 | 消息解析代码要容忍未知字段（`JsonUtility` 会忽略未知字段，这是好事） |
| 死代码 | 服务端 mongo_uri 存在但不使用 | 不要写"以后可能用到"的代码，需要时再加 |
| 构建期校验 | 服务端 BT key 运行时 panic | 配置映射（如状态→动画）应在启动时校验完整性，不要等运行时发现缺失 |
| 端到端验证 | 服务端首次 MongoDB 启动失败 | 每个功能完成后必须连接真实服务端跑一遍，不要只在本地 mock 测试 |

## WS 通信注意事项

- **world_snapshot 是高频消息**（每 100ms 一次）。解析和处理必须高效，避免 GC 压力
- **请求-响应匹配**：用 `id` 字段匹配。发送请求时生成唯一 ID，收到 response/error 时按 ID 回调
- **广播和响应混合到达**：ReadMessage 可能先收到 snapshot 再收到 response，消息分发要处理这种交错
- **断线重连**：重连后服务端不记得之前的状态，客户端需要重新 spawn 所有 NPC
